The client for this project was The Gaming Room, a company seeking to expand their existing game, Draw It or Lose It, into a web-based application that could support multiple platforms. Their primary requirement was to create a scalable, distributed system that would allow multiple teams and players to interact in real time while maintaining a consistent game state and enforcing unique naming across users and teams.

One area I performed well in was organizing the system architecture and identifying appropriate technologies for a distributed environment. I broke the requirements down into logical components such as server-side processing, client interaction, and data persistence, while also considering platform compatibility and scalability. This helped ensure the design aligned with modern cloud-based development practices.

Working through the software design document was especially helpful because it required me to think through decisions before implementation. By defining constraints, system components, and data flow early, I developed a clearer understanding of how the application should function as a whole. This reduced confusion later and made the transition from design to development more structured.

If I were to revise one part of my work, I would improve the depth of my domain model and class design. While functional, it could be expanded to better represent relationships between entities and improve clarity for future developers working on the project.

To interpret the user’s needs, I focused on the requirement for a multiplayer, cross-platform experience and translated that into a client-server architecture with centralized control. Considering the user experience is critical because the success of the application depends on usability, responsiveness, and reliability. If the system does not meet user expectations, even a technically sound design can fail.

My approach to designing software involved breaking the system into smaller components, identifying constraints, and selecting technologies that best fit the requirements. In the future, I would continue using structured design documents while also incorporating iterative design and feedback loops to refine systems earlier in the process.
